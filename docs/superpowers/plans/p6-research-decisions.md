PHASE: P6
SUMMARY: Phase 6 introduces three agent kinds (responsive / supervisor-router / monitor-loop) on top of today's single-shape `Skill` model. The roadmap text leaves the most consequential design questions explicitly open — chiefly the monitor-vs-trigger overlap, the schema shape for kind-specific config, and the supervisor's status as a graph node vs. a routing primitive. Resolving these three before planning prevents reworking the AgentKind ABC, the YAML schema, and the orchestrator's session lifecycle later.

DECISIONS:
- header: "Monitor-loop lifetime: orchestrator-level singleton vs. per-session, and whether it overlaps Phase 5's scheduled trigger"
  question: "Where does a `kind: monitor-loop` agent live in the runtime? The roadmap explicitly flags this as 'worth resolving in design': should monitor-loop be its own first-class agent kind that runs as a long-lived background daemon at the orchestrator level (one instance shared across all sessions, firing `start_session` on detected anomalies), OR should it collapse into a Phase 5 `kind: schedule` trigger pointing at a no-LLM `kind: supervisor-router` agent? The choice shapes the AgentKind ABC, lifecycle hooks (boot/shutdown), state ownership (does a monitor have a 'session' at all?), and whether the framework needs supervised-retry/cancellation primitives that Phase 5 already provides for cron triggers."
  options:
    - "Singleton daemon at orchestrator boot. `monitor_loop.py` is a separate `AgentKind`. No session until it emits. Monitor reads from MCP tools but never owns Session state; on detection it calls `orchestrator.start_session(trigger_kind='monitor', payload=...)` which then runs through the dedup pipeline (P7) and supervisor (P6). Pros: clean separation, monitor can survive across many sessions, matches ASR.md's L1-Hot-Buffer continuous polling. Cons: introduces a second long-running-task supervisor next to Phase 5's cron daemon; two retry/shutdown frameworks."
    - "Collapse into Phase 5: kill `monitor-loop` as a kind. A monitor is just a `kind: schedule` trigger with an interval, a `poll_tool`, and a `detect_rule`. The 'agent' it fires is a `kind: supervisor-router` with rule logic. Pros: one daemon framework, no new lifecycle. Cons: muddies the trigger/agent boundary; detect rules end up in trigger config which feels misplaced; ASR.md's intent of an always-on observer agent gets diluted."
    - "Hybrid: monitor-loop *is* a kind, but its lifecycle is delegated to Phase 5's schedule trigger (the framework provides the daemon supervisor; monitor-loop config declares interval + poll + detect; the AgentKind ABC has a `tick()` contract that the schedule-trigger calls). Pros: one daemon, but the user authors a real agent. Cons: tighter P5↔P6 coupling than the roadmap currently implies."
- header: "Supervisor-router as graph node vs. routing primitive (and its bookkeeping cost)"
  question: "Today every agent that runs produces an `AgentRun` record with token usage, signal, confidence, and a slot in `state.agents_run`. A supervisor-router does no LLM work — it inspects state and chooses the next node. Does it (a) become a real LangGraph node that emits an `AgentRun` (consistent bookkeeping, queryable from UI, but inflates the run history with no-op rows), (b) become a pure routing function attached to a conditional edge (zero overhead, invisible to the audit trail), or (c) record a lightweight `RouterDecision` artifact distinct from `AgentRun`? This determines whether `make_agent_node` is reused or whether `framework/graph.py` grows a parallel `make_router_node` codepath, and whether the UI's per-incident timeline needs to render routing hops."
  options:
    - "First-class node, AgentRun emitted with `tokens=0`, `model=None`, signal = chosen route. UI shows it as a dispatch step. Reuses make_agent_node skeleton; auditable. Cost: bloats `agents_run` lists for long sessions; existing token-cost dashboards need a 'router' filter."
    - "Pure conditional-edge function, no AgentRun, no graph node. The `kind: supervisor-router` skill loader registers a Python callable; `_build_agent_nodes` wires it as `add_conditional_edges(prev_agent, supervisor.route_fn, route_map)`. Zero overhead, fastest, but supervisor decisions are invisible in stored state — debugging a misroute means re-running."
    - "Distinct artifact: introduce `RouterDecision(at, from_agent, signal_in, route_out, reason)` appended to a new `state.routing_history` list. UI renders separately. Adds a state field but keeps `AgentRun` semantically pure (LLM agents only)."
- header: "Skill YAML schema: discriminated `kind` field on one Pydantic model vs. separate models per kind vs. separate registries"
  question: "Today `Skill` is a flat Pydantic model with `tools`, `routes`, `system_prompt`, `model`, `stub_response` — every field assumes responsive-LLM. A supervisor-router skill has no `model`, no `system_prompt`, no `tools`, but has `rules` (mapping signal/state → next agent). A monitor-loop skill has `interval_seconds`, `poll_tool`, `detect_rule`, no `routes`, no `model`. How should the loader represent this? The choice cascades into `load_all_skills`, the agent-name validator (does a monitor-loop have to live in `config/skills/`?), the agents listing API (`Orchestrator.list_agents`), and how validators reject illegal combinations (e.g., `kind: monitor-loop` with a `model:` field)."
  options:
    - "Single `Skill` model with `kind: Literal['responsive','supervisor-router','monitor-loop'] = 'responsive'` + a Pydantic `model_validator(mode='after')` that asserts presence/absence of fields per kind. Backward-compatible (existing skills omit `kind` and default to responsive). Field set is the union; loader stays one function. Cons: Skill model becomes wide; IDE/type narrowing weak."
    - "Discriminated union via Pydantic `Field(discriminator='kind')` with three sibling models — `ResponsiveSkill`, `SupervisorSkill`, `MonitorSkill` — sharing a `BaseSkill`. Loader returns `BaseSkill` and downstream code switches on `isinstance` or `.kind`. Cleanest typing, hardest validation by construction. Slight migration cost: existing `Skill(...)` constructions in tests need `ResponsiveSkill(...)` or kind injection."
    - "Three separate registries (`skills/`, `routers/`, `monitors/` directories), each with its own loader and Pydantic model. Maximally explicit, no field-presence validators needed. Cons: three loaders to keep in sync; agents listing API has to merge three sources; the agent-name uniqueness invariant (currently enforced inside `load_all_skills`) now needs cross-registry coordination."
